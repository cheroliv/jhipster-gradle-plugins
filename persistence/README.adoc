Plugin Gradle qui résout le problème fondamental du cycle JHipster :
régénérer le projet depuis un fichier JDL sans perdre le code métier Kotlin.

== Le problème

JHipster est un générateur d'applications Spring Boot full-stack.
Sa puissance réside dans la régénération depuis un fichier `.jdl` —
mais chaque régénération écrase l'intégralité du projet,
y compris le code métier écrit à la main.

[plantuml, problem-statement, svg]
....

skinparam backgroundColor #FEFEFE
skinparam defaultFontName "DejaVu Sans"
skinparam defaultFontSize 12

skinparam activity {
  BackgroundColor #FDFEFE
  BorderColor     #2E86C1
  FontSize        11

|Développeur|
start
:Modifie **edster.jdl**\n(nouvelle entité, relation…);
:Lance **jhipster jdl --force**;

|JHipster CLI|
:Régénère **tout** le projet;

|Développeur|
if (Code métier survit ?) then (❌ non)
  #FADBD8 :Perte irrémédiable\ndu code métier Kotlin !;

== La solution : `__codebase__/`

Le plugin introduit un répertoire sanctuaire `__codebase__/` à la racine du
projet JHipster. Tout le code métier Kotlin y est persisté.
Après chaque régénération, le plugin rsync `__codebase__/` vers le projet
régénéré — en quelques secondes, le projet est à nouveau complet.

[plantuml, solution-overview, svg]
....

  [build.gradle]

  [buildSrc/]

folder "edster/ (projet JHipster)" as jhp {

  **Sanctuaire du code métier**
  Jamais écrasé par JHipster
  Versionné indépendamment
end note

[syncCodebase] -right-> jhp : rsync\n__codebase__/ → projet
cb -right-> [syncCodebase]

@enduml

== Installation

=== Prérequis système

[cols="2,3", options="header"]
|===
| Prérequis | Détail

| JDK 17+
| Toolchain configurée automatiquement par le plugin

| nvm
| Installé dans `~/.nvm` avec l'alias `lts/jod` (Node 22)

| JHipster CLI
| `npm install -g generator-jhipster`

| Docker + Compose v2
| Pour `dockerDb`, `dockerUp`, `dev`

| rsync
| Optionnel — fallback Java natif si absent (cross-platform)
|===

sans aucune configuration explicite.

=== Référence des tâches

[cols="2,1,3", options="header"]
|===
| Tâche | Groupe | Description

| `regenerate`
| `jhipster-persistence`
| Pipeline complet : `cleanCodebase` → `generateJdl` → `syncCodebase`

| `cleanCodebase`
| `jhipster-persistence`
| Supprime les répertoires et fichiers générés par JHipster (`.gradle/`, `build/`, `src/`, `node_modules/`, etc.)

| `generateJdl`
| `jhipster-persistence`
| Lance `jhipster jdl <nom>.jdl --force` via nvm `lts/jod`. Applique les patches Kotlin post-génération.

| `syncCodebase`
| `jhipster-persistence`
| Rsync `__codebase__/` → projet régénéré (`build.gradle`, `gradle/`, `src/`, `buildSrc/`, `.github/`). Fallback Java natif si rsync absent.

| `dockerDb`
| `jhipster-persistence`
| `docker compose up -d` sur `postgresql.yml` (convention)

| `dockerUp`
| `jhipster-persistence`
| `docker compose up -d` stack complète

| `dockerDown`
| `jhipster-persistence`
| `docker compose down`

| `dev`
| `jhipster-persistence`
| `dockerDb` puis `./gradlew bootRun` dans le projet JHipster cible
|===

== Utilisation courante

=== Modifier le modèle de données

[source,bash]
----
# 1. Éditer le fichier JDL
vim edster/edster.jdl

# 2. Régénérer (clean → generate → sync)
./gradlew regenerate

# 3. Démarrer en mode développement
./gradlew dev
----

=== Synchroniser uniquement (sans régénérer)

[source,bash]
----
# Après avoir modifié __codebase__/ manuellement
./gradlew syncCodebase
----

=== Gérer Docker séparément

[source,bash]
----
# Démarrer PostgreSQL uniquement
./gradlew dockerDb

# Démarrer toute la stack (PostgreSQL + Elasticsearch + etc.)
./gradlew dockerUp

# Arrêter tous les services
./gradlew dockerDown
----

== Manifest de synchronisation (`SyncCodebaseTask`)

Les artefacts synchronisés depuis `__codebase__/` vers le projet JHipster :

[cols="2,2,1", options="header"]
|===
| Source dans `__codebase__/` | Destination dans le projet | Type

| `build.gradle`
| `./build.gradle`
| Fichier

| `gradle/`
| `./gradle/`
| Répertoire

| `src/`
| `./src/`
| Répertoire

| `.github/`
| `./.github/`
| Répertoire

| `buildSrc/`
| `./buildSrc/`
| Répertoire
|===

TIP: `build/` et `.gradle/` sont nettoyés avant la synchronisation
pour éviter les conflits de cache Gradle.

== Patches post-génération (`GenerateJdlTask`)

Après chaque régénération JHipster, ces patches sont appliqués automatiquement :

[cols="2,3", options="header"]
|===
| Fichier patché | Modification

| `.gitignore`
| Ajoute `.goose`, `README.pdf`, `README.html`, `README.docx`, `README.epub` (fr et en)

| `gradle/libs.versions.toml`
| Remplace `kotlin = "x.y.z"` par `kotlin = "2.3.20"`

| `build.gradle`
| Injecte `kotlin.jvmToolchain(17)` (idempotent via marqueur de commentaire)

| `buildSrc/build.gradle`
| Met à jour `kotlin-gradle-plugin:x.y.z` vers `2.3.20`
|===

== Référence des erreurs courantes

[cols="2,3", options="header"]
|===
| Erreur | Solution

| `Aucun projet JHipster trouvé`
| Vérifier que le projet frère contient `.yo-rc.json`

| `Fichier JDL introuvable`
| Le fichier JDL doit s'appeler `<nom-du-projet>.jdl`

| `nvm requis`
| Installer nvm et créer l'alias `nvm alias lts/jod lts/jod`

| `jhipster: command not found`
| `npm install -g generator-jhipster` avec nvm `lts/jod` actif

| `__codebase__ introuvable`
| Créer le répertoire manuellement ou lancer `syncCodebase` après `generateJdl`
|===
